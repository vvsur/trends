# Концепция корпоративного портала трендов

## Видение

Корпоративный портал трендов - единая рабочая среда для управления технологическими, биржевыми, финансовыми, регуляторными, HR, продуктовыми и операционными трендами, а также связанными с ними инновациями, гипотезами и пилотами. Портал переводит работу с трендами из ручных презентаций и разрозненных таблиц в измеримый PDCA-процесс.

## Цели продукта

- Создать прозрачный технологический радар компании.
- Расширить радар до корпоративной карты трендов: технологии, финансы и биржевая инфраструктура, клиенты, регуляторика, HR, операционная устойчивость, ESG.
- Обеспечить быстрый и единый скоринг инноваций.
- Дать руководителям видимость KPI 1.3 и прогресса по платформам/департаментам.
- Ускорить переход от идеи к пилоту и от пилота к решению об эксплуатации.
- Собирать библиотеку проверенных бизнес-кейсов.
- Сделать ограничения ИБ, архитектуры, инфраструктуры и ресурсов видимыми на ранней стадии.
- Обеспечить регулярное обновление трендов из проверенных внешних источников через безопасный импорт из открытого контура в закрытый.

## Пользователи

| Роль | Потребности |
| --- | --- |
| Руководитель ИТ/департамента | Видеть KPI, прогресс, блокеры, эффект, решения на квартал |
| Владелец тренда | Вести карточку тренда, гипотезы, источники, актуальность |
| Автор идеи | Предложить инновацию, видеть статус и обратную связь |
| Эксперт/скоринговая комиссия | Оценить impact/confidence/ease, риски и ограничения |
| Владелец пилота | Управлять пилотом, метриками успеха, LT/CT, итогами |
| ИБ/архитектура/инфраструктура | Раннее согласование ограничений и требований |
| HR/коммуникации | Обучение, PR инновационной культуры, кейсы успеха |
| Доменный эксперт | Проверять внешние сигналы, подтверждать релевантность трендов, связывать их с инициативами |
| Администратор источников | Управлять реестром проверенных источников, правилами сбора и trust level |

## Основные сущности

| Сущность | Смысл |
| --- | --- |
| Trend | Технологическое, биржевое, финансовое, HR, регуляторное, продуктовое или другое значимое направление изменений |
| Signal | Источник/событие, подтверждающее или ослабляющее тренд |
| Innovation | Конкретная инновационная идея или возможность |
| Hypothesis | Проверяемое предположение с ожидаемым эффектом |
| Scoring | Быстрая оценка и ранжирование |
| Pilot | Ограниченная проверка гипотезы |
| Metric | Целевой или фактический показатель |
| Decision | Управленческое решение по итогам оценки |
| Business Case | Апробированный кейс для повторного применения |
| Risk/Constraint | Ограничение ИБ, архитектуры, инфраструктуры, ресурсов, compliance |
| Source | Проверенный внешний или внутренний источник сигналов |
| Import Batch | Подписанный пакет внешних сигналов, переданный из открытого контура |

## Workflow

1. Открытый сборщик получает публичные сигналы из проверенных источников и формирует безопасный пакет.
2. Закрытый портал импортирует пакет в staging, проверяет подпись, hash, схему и источник.
3. Доменный эксперт подтверждает сигнал, связывает его с существующим трендом или создает новый тренд.
4. Автор или владелец тренда добавляет инновацию/гипотезу на основе тренда.
5. Инновация связывается с трендом, департаментом, владельцем и источниками.
6. Эксперты проводят быстрый скоринг.
7. Приоритетные инновации переводятся в гипотезы и пилоты.
8. Для пилота задаются критерии успеха, LT/CT, метрики, ресурсы и ограничения.
9. По итогам пилота фиксируется решение: масштабировать, внедрить, остановить, отложить, пересобрать.
10. Успешные результаты попадают в библиотеку бизнес-кейсов.
11. Дашборды обновляют KPI и материалы для ежемесячного/квартального контура.

## Модули портала

### 1. Корпоративный радар трендов

- Визуализация трендов по доменам и зрелости.
- Режимы: список, радар, матрица impact/effort, timeline.
- Кольца: Adopt, Trial, Assess, Hold или локализованные аналоги "Внедрять", "Пилотировать", "Изучать", "Пауза".
- Фильтры по департаменту, владельцу, домену, году, статусу, актуальности.
- Домены: технологии, биржевая и финансовая инфраструктура, клиенты и продукты, регуляторика, HR, операционная устойчивость, макро/отрасль, ESG.

### 2. Реестр инноваций

- Единый список инициатив из материалов и новых предложений.
- Статусы pipeline.
- Ответственные, департаменты, даты, комментарии.
- Связи с трендами, гипотезами, пилотами и кейсами.

### 3. Скоринг

- Единая форма оценки.
- ICE + корпоративные ограничители.
- История оценок.
- Сравнение по платформам/департаментам.
- Рекомендация к пилоту или отказу.

### 4. Пилоты

- Карточка пилота.
- Цели и критерии успеха.
- LT/CT.
- Риски, зависимости, решения ИБ/архитектуры.
- Итоговый отчет и решение.

### 5. Дашборды KPI

- Скоринг >= 3 инноваций на платформу/департамент в квартал.
- Доля платформ/департаментов с минимум одним пилотом актуального тренда.
- Соблюдение дорожной карты >= 80%.
- Доля инноваций с замеренным LT/CT.
- Доля нереализуемых инноваций с причинами.
- Количество внедренных/масштабированных пилотов.

### 6. Библиотека кейсов

- Проверенные бизнес-кейсы.
- Повторное использование.
- Контакты владельцев.
- Фактический эффект и ограничения.

### 7. Управленческий контур

- Ежемесячная повестка.
- Квартальные итоги.
- Список решений на комитет.
- Экспорт в презентацию/отчет.

### 8. Обновление из внешних источников

- Реестр проверенных источников по доменам.
- Открытый сборщик информации вне закрытого контура.
- Подписанные пакеты сигналов в формате JSON/CSV/Markdown.
- Staging, проверка источника, hash, схемы и ручное подтверждение доменным экспертом.
- История импортов и audit log.

## MVP

MVP должен покрыть 2026 KPI и создать основу для 2027 PDCA:

- карточка тренда;
- домены трендов за пределами технологии;
- реестр инноваций;
- быстрый скоринг;
- pipeline статусов;
- карточка пилота;
- базовые KPI-дашборды;
- импорт текущего реестра из PDF;
- ручной/полуавтоматический импорт внешних сигналов из проверенных источников;
- роли владельца, эксперта, руководителя и администратора.

## Не-MVP

- автоматический сбор сигналов из внешних источников;
- AI-генерация прогнозных инноваций;
- сложное портфельное бюджетирование;
- глубокая интеграция с Jira/Confluence/ServiceNow;
- автоматический экспорт презентаций;
- marketplace кейсов.
- полностью автоматическая публикация трендов без review;
- двусторонняя синхронизация закрытого портала с интернетом.

Эти функции логичны для горизонта 2027-2028, после стабилизации базового процесса.

## Архитектура обновления

Целевая архитектура разделяет сбор и внутреннюю работу:

- открытый контур собирает публичные данные из проверенных источников;
- закрытый контур хранит внутренние тренды, инициативы, скоринг, пилоты и KPI;
- между контурами передаются только нормализованные подписанные пакеты без активного содержимого;
- публикация тренда происходит только после review доменного эксперта.
