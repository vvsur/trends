# Концепция корпоративного портала трендов

## Видение

Корпоративный портал трендов - единая рабочая среда для управления технологическими, биржевыми, финансовыми, регуляторными, HR, продуктовыми и операционными трендами, а также связанными с ними инновациями, гипотезами и пилотами. Портал переводит работу с трендами из ручных презентаций и разрозненных таблиц в измеримый PDCA-процесс.

## Цели продукта

- Создать прозрачный технологический радар компании.
- Расширить радар до корпоративной карты трендов: технологии, финансы и биржевая инфраструктура, клиенты, регуляторика, HR, операционная устойчивость, ESG.
- Обеспечить быстрый и единый скоринг инноваций.
- Дать руководителям видимость KPI 1.3 и прогресса по платформам/департаментам.
- Ускорить переход от идеи к пилоту и от пилота к решению об эксплуатации.
- Собирать библиотеку проверенных бизнес-кейсов.
- Сделать ограничения ИБ, архитектуры, инфраструктуры и ресурсов видимыми на ранней стадии.
- Обеспечить регулярное обновление трендов из проверенных внешних источников через безопасный импорт из открытого контура в закрытый.
- Связать портал с измеримой ценностью для компании: выручкой, снижением затрат, управлением рисками, рыночной позицией, рейтингом работодателя и репутацией.
- Сделать портал полезным лично для сотрудников: показывать влияние трендов на роль, навыки, карьеру, пилоты и видимость вклада.

## Пользователи

| Роль | Потребности |
| --- | --- |
| Руководитель ИТ/департамента | Видеть KPI, прогресс, блокеры, эффект, решения на квартал |
| Владелец тренда | Вести карточку тренда, гипотезы, источники, актуальность |
| Автор идеи | Предложить инновацию, видеть статус и обратную связь |
| Эксперт/скоринговая комиссия | Оценить impact/confidence/ease, риски и ограничения |
| Владелец пилота | Управлять пилотом, метриками успеха, LT/CT, итогами |
| ИБ/архитектура/инфраструктура | Раннее согласование ограничений и требований |
| HR/коммуникации | Обучение, PR инновационной культуры, кейсы успеха |
| Доменный эксперт | Проверять внешние сигналы, подтверждать релевантность трендов, связывать их с инициативами |
| Администратор источников | Управлять реестром проверенных источников, правилами сбора и trust level |
| Сотрудник любой роли | Понимать, какие тренды влияют на его работу, какие навыки развивать, где можно предложить идею или присоединиться к пилоту |

## Основные сущности

| Сущность | Смысл |
| --- | --- |
| Trend | Технологическое, биржевое, финансовое, HR, регуляторное, продуктовое или другое значимое направление изменений |
| Signal | Источник/событие, подтверждающее или ослабляющее тренд |
| Innovation | Конкретная инновационная идея или возможность |
| Hypothesis | Проверяемое предположение с ожидаемым эффектом |
| Scoring | Быстрая оценка и ранжирование |
| Pilot | Ограниченная проверка гипотезы |
| Metric | Целевой или фактический показатель |
| Decision | Управленческое решение по итогам оценки |
| Business Case | Апробированный кейс для повторного применения |
| Risk/Constraint | Ограничение ИБ, архитектуры, инфраструктуры, ресурсов, compliance |
| Source | Проверенный внешний или внутренний источник сигналов |
| Import Batch | Подписанный пакет внешних сигналов, переданный из открытого контура |

## Workflow

1. На первом этапе владелец тренда или эксперт вручную добавляет IT-тренд, инновацию или гипотезу.
2. Данные можно загрузить из PDF/таблицы, но после загрузки они доступны для ручной корректировки.
3. На следующих этапах открытый сборщик получает публичные сигналы из проверенных источников и формирует безопасный пакет.
4. Закрытый портал импортирует пакет в staging, проверяет подпись, hash, схему и источник.
5. Доменный эксперт подтверждает сигнал, связывает его с существующим трендом или создает новый тренд.
6. Инновация связывается с трендом, департаментом, владельцем и источниками.
7. Эксперты проводят быстрый скоринг.
8. Приоритетные инновации переводятся в гипотезы и пилоты.
9. Для пилота задаются критерии успеха, LT/CT, метрики, ресурсы и ограничения.
10. По итогам пилота фиксируется решение: масштабировать, внедрить, остановить, отложить, пересобрать.
11. Успешные результаты попадают в библиотеку бизнес-кейсов.
12. Дашборды обновляют KPI и материалы для ежемесячного/квартального контура.

## Принцип ручного управления

Ручной ввод и ручная корректировка должны быть доступны на всех этапах развития системы. Автоматический сбор, импорт, классификация и AI-summary не являются источником окончательной истины.

Обязательные правила:

- пользователь с правами может вручную создать и изменить тренд, сигнал, источник, инновацию, скоринг, пилот, метрику, решение и связь между сущностями;
- импортированные данные попадают в staging и могут быть отредактированы до публикации;
- опубликованные данные можно исправить вручную с указанием причины изменения;
- автоматические обновления не должны затирать ручные правки без подтверждения владельца;
- все ручные и автоматические изменения сохраняются в audit log;
- для управленческих решений всегда фиксируется человек, принявший решение.

## Этапы развития

| Этап | Фокус | Способ наполнения | Результат |
| --- | --- | --- | --- |
| 1 | IT/технологические тренды | Ручной ввод, импорт PDF/таблиц, экспертная корректировка | Закрытие KPI 1.3 и запуск PDCA по технологическим инновациям |
| 2 | Биржевые и финансовые тренды | Ручной ввод + проверенные внешние источники по FMI/рынкам/регуляторам | Связь рыночных изменений с продуктами, процессами и IT-гипотезами |
| 3 | Регуляторика, устойчивость, киберриски | Ручной ввод, alert-сигналы, review доменных экспертов | Раннее выявление обязательных изменений и ограничений |
| 4 | HR, организация, культура | Ручной ввод, HR-исследования, внутренние данные | Связь трендов с навыками, обучением и коммуникациями |
| 5 | Клиенты, продукты, макро/отрасль, ESG | Ручной ввод, отраслевые источники, клиентские исследования | Полная корпоративная карта трендов |
| 6 | Автоматизация open collector | Подписанные batch-пакеты из открытого контура | Снижение ручного поиска без потери экспертного контроля |
| 7 | AI-assisted analytics | AI-черновики и прогнозные гипотезы с review | Прогнозные инновации и сценарный анализ |

Подробная дорожная карта: `docs/08-implementation-roadmap.md`.

## Модули портала

### 1. Корпоративный радар трендов

- Визуализация трендов по доменам и зрелости.
- Режимы: список, радар, матрица impact/effort, timeline.
- Кольца: Adopt, Trial, Assess, Hold или локализованные аналоги "Внедрять", "Пилотировать", "Изучать", "Пауза".
- Фильтры по департаменту, владельцу, домену, году, статусу, актуальности.
- Домены: технологии, биржевая и финансовая инфраструктура, клиенты и продукты, регуляторика, HR, операционная устойчивость, макро/отрасль, ESG.

### 2. Реестр инноваций

- Единый список инициатив из материалов и новых предложений.
- Статусы pipeline.
- Ответственные, департаменты, даты, комментарии.
- Связи с трендами, гипотезами, пилотами и кейсами.

### 3. Скоринг

- Единая форма оценки.
- ICE + корпоративные ограничители.
- История оценок.
- Сравнение по платформам/департаментам.
- Рекомендация к пилоту или отказу.

### 4. Пилоты

- Карточка пилота.
- Цели и критерии успеха.
- LT/CT.
- Риски, зависимости, решения ИБ/архитектуры.
- Итоговый отчет и решение.

### 5. Дашборды KPI

- Скоринг >= 3 инноваций на платформу/департамент в квартал.
- Доля платформ/департаментов с минимум одним пилотом актуального тренда.
- Соблюдение дорожной карты >= 80%.
- Доля инноваций с замеренным LT/CT.
- Доля нереализуемых инноваций с причинами.
- Количество внедренных/масштабированных пилотов.

### 6. Библиотека кейсов

- Проверенные бизнес-кейсы.
- Повторное использование.
- Контакты владельцев.
- Фактический эффект и ограничения.

### 7. Управленческий контур

- Ежемесячная повестка.
- Квартальные итоги.
- Список решений на комитет.
- Экспорт в презентацию/отчет.

### 8. Обновление из внешних источников

- Реестр проверенных источников по доменам.
- Открытый сборщик информации вне закрытого контура.
- Подписанные пакеты сигналов в формате JSON/CSV/Markdown.
- Staging, проверка источника, hash, схемы и ручное подтверждение доменным экспертом.
- История импортов и audit log.

### 9. Личная польза для сотрудников

- Персональная страница "Мои тренды" по роли, подразделению, навыкам и подпискам.
- Влияние тренда на сотрудника: новые навыки, автоматизация, риски, возможности, открытые пилоты.
- Профиль вклада: идеи, review, пилоты, кейсы, экспертные комментарии.
- Обратная связь по идеям с SLA и причиной решения.
- Рекомендации по обучению и внутренним экспертам.

### 10. Value scorecard компании

- Выручка и pipeline от трендов и пилотов.
- Экономия затрат и рост производительности.
- Снижение рисков и раннее выявление ограничений.
- Рыночная позиция, публичная инновационная видимость, вклад в рейтинг работодателя.
- Вовлеченность сотрудников, повторные участники, скорость обратной связи по идеям.

### 11. Design System / UI Rules

- Источник корпоративной дизайн-системы: `.skills`, подключаемый как git submodule из `git@github.com:ui-sigma/sigma-skills.git`.
- Обязательные скиллы для будущей UI-разработки: `foundations-tokens`, `foundations-themes`, `component-button`.
- Все UI-изменения должны использовать токены, темы, states и button recipes из MOEX design system.
- Запрещены ad hoc UI-стили: самодельные цвета, состояния, границы, тени, варианты кнопок, raw-цвета в CSS/TS/TSX вне токенов.
- Интерфейс должен поддерживать light, dark и inverted темы.
- После UI-изменений обязательна браузерная проверка на текущем разрешении.
- Подробно: `docs/10-design-system-ui-rules.md`.

### 12. Technology Stack

- Frontend должен быть реализован на React.
- Backend должен быть реализован на TypeScript.
- Для MVP выбран storage baseline: SQLite + Prisma ORM/Migrate.
- PostgreSQL не используется в MVP и может быть рассмотрен позже только через отдельное архитектурное решение и план миграции.
- Конкретные framework/runtime поверх React и TypeScript выбираются отдельным архитектурным решением до начала реализации.
- Подробно: `docs/11-technology-stack.md`.

## MVP

MVP должен покрыть 2026 KPI и создать основу для 2027 PDCA:

- карточка тренда;
- фокус на IT/технологических трендах;
- персональная страница "Мои тренды";
- профиль вклада сотрудника;
- SLA обратной связи по идеям;
- реестр инноваций;
- формулирование проверяемой гипотезы из инновации;
- ранняя фиксация ограничений ИБ, архитектуры, инфраструктуры, данных, ресурсов и compliance;
- быстрый скоринг;
- pipeline статусов;
- карточка пилота;
- базовые KPI-дашборды;
- базовый value scorecard для компании;
- ручной ввод трендов, инноваций, скорингов и пилотов;
- ручная корректировка всех данных с audit log;
- импорт текущего реестра из PDF;
- роли владельца, эксперта, руководителя и администратора;
- подключение MOEX design system skills и фиксация правил UI-разработки в `AGENTS.md` до начала активной UI-реализации;
- хранение данных MVP на SQLite через Prisma ORM/Migrate;
- соблюдение ограничений стека: React на frontend и TypeScript на backend.

## Не-MVP

- автоматический сбор сигналов из внешних источников;
- полноценное подключение биржевых, HR, ESG и прочих нетехнологических доменов;
- AI-генерация прогнозных инноваций;
- сложное портфельное бюджетирование;
- глубокая интеграция с Jira/Confluence/ServiceNow;
- автоматический экспорт презентаций;
- marketplace кейсов.

Эти функции логичны для горизонта 2027-2028, после стабилизации базового процесса.

## Запрещенные направления без отдельного решения

- миграция UI без предварительного подключения MOEX design system и правил в `AGENTS.md`;
- полностью автоматическая публикация трендов без review;
- двусторонняя синхронизация закрытого портала с интернетом.

## Архитектура обновления

Целевая архитектура разделяет сбор и внутреннюю работу:

- открытый контур собирает публичные данные из проверенных источников;
- закрытый контур хранит внутренние тренды, инициативы, скоринг, пилоты и KPI;
- между контурами передаются только нормализованные подписанные пакеты без активного содержимого;
- публикация тренда происходит только после review доменного эксперта.

Подробно: `docs/06-trend-domains-and-refresh.md` и `docs/07-architecture.md`.

## Ценность для компании и сотрудников

Портал должен измеряться не только количеством трендов и пилотов. Для компании он должен показывать вклад в выручку, эффективность, снижение рисков, рыночную позицию и репутацию. Для сотрудников он должен показывать личную пользу: влияние трендов на работу, развитие навыков, доступ к пилотам, обратную связь по идеям и видимость вклада.

Подробно: `docs/09-independent-value-analysis.md`.
